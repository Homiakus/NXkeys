# NX Pro Hybrid для Siemens NX 2512

Этот документ задаёт актуальные принципы профиля NXKeys, модульного Leader-меню, радиальных меню и целевого покрытия высокочастотных операций.

Целевая среда:

- Siemens NX / Designcenter NX 2512;
- Windows x64;
- механическое проектирование;
- профили `nx2512-pro-hybrid.json` и `nx2512-ergo-80.json`;
- HotkeyStudio, Control Center и Command Bridge.

## 1. Цель профиля

Профиль должен давать быстрый доступ минимум к 80% операций типового рабочего процесса конструктора, а не к 80% всех команд NX.

NX содержит тысячи команд и специализированные приложения. Попытка назначить отдельное сочетание каждой команде ухудшает запоминаемость и увеличивает число конфликтов.

Целевой поток:

```text
создание/открытие
→ эскиз
→ базовая форма
→ отверстия и кромки
→ паттерны и симметрия
→ редактирование
→ сборка
→ PMI/чертёж
→ измерение и проверка
→ повторное использование
```

## 2. Уровни доступа

| Уровень | Назначение |
|---|---|
| Штатные сочетания NX | файлы, отмена, вид, системные команды |
| Небольшой банк глобальных сочетаний | наиболее частые операции |
| Leader-меню | большое число команд без конфликтов клавиатуры |
| Application/Object Radial | команды рядом с курсором и выбранным объектом |
| Command Finder/API Explorer | редкие и исследовательские команды |

## 4. Модульное Leader-меню

Активное приложение NX выбирает видимый набор команд.

Основные модули:

```text
modeling
sketch
assembly
drafting
pmi
surface
sheet_metal
manufacturing
simulation
routing
mold
reuse
inspect_view
selection_object
```

Общие слои `selection_object`, `inspect_view` и `reuse` могут быть доступны поверх основного модуля.

## 5. Постоянная семантика направлений

| Слот | Смысл |
|---|---|
| `N` | начать, создать или открыть основной объект |
| `NE` | следующий основной шаг |
| `E` | добавить объект, материал или зависимость |
| `SE` | преобразовать или заменить |
| `S` | завершить, удалить или выполнить вторичную обработку |
| `SW` | убрать, ослабить или уменьшить |
| `W` | структура, связь или размножение |
| `NW` | проверка, измерение или служебная команда |

Смысл направления должен оставаться похожим во всех модулях, даже когда конкретные команды различаются.

## 6. Управление Leader HUD

| Клавиша | Действие |
|---|---|
| trigger | открыть Leader |
| `Tab` / `Shift+Tab` | переключить модуль вручную |
| `Space` | перейти в поиск |
| `Backspace` | удалить символ или вернуться |
| `Esc` | отменить и закрыть HUD |
| `Enter` | подтвердить опасную команду |

Рекомендуемый trigger — `CapsLock`, если он не используется пользователем для переключения регистра.

## 7. Правила последовательностей

1. Последовательность должна быть короткой и мнемоничной.
2. Первая клавиша обычно обозначает модуль или категорию.
3. Последовательности не должны конфликтовать после удаления пробелов и нормализации регистра.
4. Часто используемые команды получают более короткий путь.
5. Опасная команда не должна выполняться без подтверждения.
6. Команда, требующая выбранный объект, помечается `requires_selection`.
7. В производственном профиле задаётся точный `BUTTON ID`.
8. Имя и aliases используются для поиска и отображения.

Пример:

```json
{
  "sequence": "M E",
  "category": "Modeling",
  "module_id": "modeling",
  "enabled": true,
  "command": {
    "id": "UG_MODELING_EXTRUDE",
    "name": "Extrude",
    "aliases": ["Выдавливание"]
  },
  "requires_selection": false,
  "destructive": false,
  "confirm_before_execute": false
}
```

## 8. Контекстная адаптация

Control Center оценивает команды по следующим сигналам:

- совпадение модуля;
- принадлежность общему слою;
- требование выбора;
- известное состояние рабочей детали;
- известное модальное окно;
- разрушительность;
- частота и давность использования.

Состояние `unknown` не должно интерпретироваться как строгий запрет. Например, `selection_count: -1` означает отсутствие данных, а не нулевой выбор.

HotkeyStudio Leader HUD в текущей реализации сохраняет собственную модульную логику и индексы последовательностей. Полное объединение ранжирования Control Center и HUD является отдельным этапом развития.

## 12. Drafting и PMI

Целевой минимум:

- создание/открытие чертежа;
- базовый вид;
- проекционный вид;
- разрез;
- размеры;
- допуски и обозначения;
- centerline/center mark;
- balloon и parts list;
- PMI dimension/annotation;
- обновление видов.

## 13. Специализированные приложения

Для CAM, CAE, Routing, Mold и Sheet Metal создаются отдельные модули. Они не включаются в общий показатель 80% механического CAD-потока без отдельной модели частот.

Опасными обычно считаются:

- запуск расчёта;
- генерация/постпроцессинг траектории;
- удаление операций;
- замена компонентов и маршрутов;
- обновление больших зависимых структур.

## 14. Радиальные меню

Используются два класса:

- Application Radial — зависит от активного приложения;
- Object Radial — зависит от типа выбранного объекта.

NXKeys хранит намерение radial в JSON и формирует план. Для автоматического развёртывания требуется роль `.mtx`, экспортированная и проверенная в целевой версии NX.

Внутренности `.mtx` не редактируются.

## 15. Покрытие

Рекомендуется вести несколько метрик.

| Метрика | Определение |
|---|---|
| Workflow coverage | взвешенная доля типовых пользовательских операций, доступных коротким путём |
| Leader profile coverage | доля запланированных операций, имеющих Leader-последовательность |
| Verified ID coverage | доля включённых команд с точным `BUTTON ID` |
| Catalog coverage | доля ID, найденных в меню целевой установки |
| Runtime coverage | доля команд, успешно принятых NX в нужном контексте |
| API candidate coverage | доля команд с найденным NXOpen/UFUN-кандидатом |

Формула workflow-покрытия:

```text
сумма весов покрытых операций
────────────────────────────── × 100%
сумма весов целевых операций
```

Вес должен основываться на реальных сценариях роли пользователя, а не на количестве команд NX.

## 16. Критерий «команда покрыта»

Для строгой оценки операция считается покрытой, когда:

1. определён пользовательский сценарий;
2. назначен доступный путь: штатная клавиша, Leader, radial или поиск;
3. указан точный ID либо однозначно подтверждён другой механизм;
4. команда найдена в целевой версии NX;
5. известны требования модуля и выбора;
6. опасность и подтверждение настроены;
7. выполнен runtime-тест;
8. ошибка имеет понятное сообщение и безопасный отказ.

## 17. API-каталог

Catalog Studio формирует инвентаризацию UI/NXOpen/UFUN. Control Center выполняет локальный поиск.

Crosswalk UI → API является эвристикой, потому что:

- одна кнопка может запускать сложный диалог;
- диалог может использовать несколько Builder/Manager;
- один API может не иметь UI-кнопки;
- сигнатуры зависят от версии;
- правильная последовательность вызовов важнее совпадения имени.

## 18. Проверка профиля

Перед применением:

```powershell
NX2512_HotkeyStudio.exe validate --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe scan --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe plan --config .\config\nx2512-pro-hybrid.json
```

После применения:

```powershell
NX2512_HotkeyStudio.exe health --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe bridge-status
```

## 19. Правила сопровождения

- не переносить ID между версиями NX без проверки;
- не считать английское имя стабильным идентификатором;
- хранить эталонный JSON в Git;
- не добавлять локальные runtime-файлы;
- повторно тестировать профиль после смены роли, лицензии или NX Maintenance Pack;
- отдельно тестировать опасные операции;
- сохранять проверенную роль `.mtx` только как непрозрачный артефакт;
- измерять фактическую частоту операций перед расширением раскладки.
