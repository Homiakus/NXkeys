# Модель безопасности NXKeys

NXKeys предназначен для обратимой настройки Siemens NX 2512 без изменения файлов установки Siemens и без неявного вмешательства во внутренние форматы ролей.

## 1. Основные инварианты

1. Файлы установки Siemens не перезаписываются.
2. Неизвестные и непрозрачные бинарные форматы не патчатся.
3. Перед первой записью целевой файл помещается в резервную копию.
4. При `atomic_writes: true` запись выполняется во временный файл с последующей заменой в том же каталоге.
5. Для одинакового JSON-профиля и результата сканирования формируется детерминированный план.
6. Неразрешённые и неоднозначные команды исключаются из генерируемого оверлея.
7. Восстановление проверяет хеш состояния после применения перед заменой или удалением файла.
8. Разрушительные Leader-команды должны быть явно помечены и подтверждаться пользователем.
9. Command Bridge не выполняет кнопку, если NX сообщает, что она отсутствует, недоступна или нечувствительна.
10. Запрос Bridge имеет ограниченный срок действия и не должен считаться бессрочным намерением пользователя.

## 2. Что NXKeys изменяет

В зависимости от режима NXKeys может создавать или обновлять:

- приватный `custom_dirs.dat`;
- `.men`-оверлеи;
- `.tbr` и `.rtb` размещения;
- скрипты запуска;
- копию проверенной роли `.mtx`;
- JSON-профиль;
- runtime-очереди и журналы в `%LOCALAPPDATA%\NXKeys`.

## 3. Что NXKeys не изменяет

- исполняемые файлы и библиотеки Siemens;
- штатные MenuScript-файлы установки NX;
- внутреннюю структуру `user.mtx` и других `.mtx`;
- лицензии и настройки сервера лицензирования;
- базы Teamcenter;
- модели, чертежи и другие пользовательские данные NX напрямую.

Команда, запущенная через Bridge, может изменить открытую модель так же, как ручное нажатие соответствующей кнопки. За последствия самой команды отвечает текущий документ NX и пользовательский контекст.

## 4. Обрабатываемые угрозы

- прерывание записи;
- повреждение целевого файла при частичной записи;
- повторное добавление пользовательского каталога;
- несовместимое или локализованное имя команды;
- неоднозначное сопоставление имени;
- изменение файла пользователем после применения NXKeys;
- отсутствующий путь к NX;
- замена существующей роли или конфигурации;
- работа NX во время обновления загруженной DLL;
- устаревший запрос Bridge;
- попытка выполнить недоступный `BUTTON ID`;
- ошибочная версия `.men`, `.tbr` или `.rtb`.

## 5. Ограничения, которые нельзя полностью устранить вне NX

- конфликты ускорителей внутри непрозрачной роли `.mtx`;
- доступность команды для конкретной лицензии;
- команды, не представленные как MenuScript `BUTTON`;
- семантическое изменение одного `BUTTON ID` между версиями NX;
- внутреннее устройство radial без роли, экспортированной из целевой версии NX;
- корректность кандидатного сопоставления UI-команды с NXOpen или UFUN;
- состояние активного диалога и выбора, если установленная версия Bridge не публикует эти поля;
- последствия выполнения команды в пользовательской модели.

## 6. Уровни доверия к команде

| Уровень | Значение |
|---|---|
| Имя | используется только для поиска и отображения |
| Кандидат | найдено вероятное совпадение по имени или API-crosswalk |
| Точный ID | в профиле указан конкретный `BUTTON ID` |
| Найден в каталоге | ID обнаружен в меню целевой установки |
| Принят NX | Bridge нашёл кнопку и NX сообщил, что она доступна |
| Выполнен | NX принял вызов кнопки и Bridge записал успешный результат |

Наличие точного ID не равно гарантии выполнения. Самый сильный сигнал — успешное выполнение в нужном модуле и контексте.

## 7. Опасные операции

Команды удаления, замены компонентов, запуска расчётов, постпроцессинга, перегенерации и другие потенциально необратимые действия должны иметь:

```json
{
  "destructive": true,
  "confirm_before_execute": true
}
```

Leader HUD требует `Enter` для подтверждения. `Esc` отменяет действие.

## 8. Резервное копирование

Резервные копии располагаются в:

```text
%LOCALAPPDATA%\NXKeys\backups
```

Каждая операция должна содержать манифест с:

- исходным и целевым путём;
- временем;
- SHA-256;
- информацией о создании, замене или удалении;
- состоянием, ожидаемым при восстановлении.

Не удаляйте резервную копию до проверки запуска NX и профиля.

## 9. Безопасное применение

Рекомендуемый порядок:

```powershell
NX2512_HotkeyStudio.exe validate --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe scan --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe plan --config .\config\nx2512-pro-hybrid.json
NX2512_HotkeyStudio.exe apply --config .\config\nx2512-pro-hybrid.json --yes
NX2512_HotkeyStudio.exe health --config .\config\nx2512-pro-hybrid.json
```

Перед организационным развёртыванием:

1. используйте тестовую рабочую станцию;
2. применяйте управляемый wrapper;
3. проверяйте все высокочастотные команды;
4. отдельно проверяйте опасные команды;
5. повторяйте тесты для каждой роли, лицензии и сборки NX;
6. сохраняйте проверенный JSON и экспортированную роль в системе контроля версий.

## 10. Реагирование на ошибку

При проблеме:

1. прекратите повторное применение;
2. закройте NX;
3. сохраните журналы и `bridge/failed`;
4. выполните `health`;
5. сравните план и установленный пакет;
6. восстановите последний манифест при подтверждённом повреждении;
7. не удаляйте исходные журналы до определения причины.
