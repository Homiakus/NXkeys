# Модель безопасности NXKeys

## Принцип наименьшего количества сочетаний

NXKeys устанавливает только 12 базовых глобальных ускорителей. Все остальные операции проходят через адаптивный контекстный путь. Это уменьшает конфликты с ролями NX и исключает глобальное назначение команд, корректность которых зависит от приложения или выбора.

## Контекстный scope

Профессиональная команда видима только когда её `module_id` совпадает с модулем, определённым из свежего контекста Bridge.

Если:

- `context.json` отсутствует;
- контекст устарел;
- Bridge не имеет статуса `running`;
- приложение не сопоставлено с профилем;

Leader не активирует рабочий набор.

Команда другого модуля блокируется. Скрытая автоматическая смена приложения запрещена. Явная смена выполняется только через `Tab`/`Shift+Tab` и подтверждается новым контекстом.

## Guards

До dispatch проверяются:

- module/application;
- freshness и confidence;
- modal dialog;
- work/display part;
- selection count;
- selection types;
- destructive/confirmation policy.

Bridge повторяет критические проверки по ожидаемой ревизии, ожидаемому количеству выбранных объектов и ожидаемому приложению.

## Подтверждение

`destructive=true` или `confirm_before_execute=true` всегда переводят HFSM в `AwaitingConfirmation`. Только `Enter` формирует запрос с `confirmation_accepted=true`.

Примеры:

- Replace/Remove Component;
- Postprocess/Delete Operation;
- Solve/Delete Simulation Object;
- Delete/Remove Route Object;
- Replace Feature Template.

## Надёжная очередь

```text
pending → processing → completed | failed
```

Гарантии:

- атомарная запись запроса;
- атомарный захват в `processing`;
- уникальный `request_id`;
- ограниченный срок действия;
- отсутствие автоматического повтора после возможного исполнения;
- `interrupted_unknown` после прерывания NX;
- результат хранится отдельно от запроса.

## Deployment

NXKeys не должен:

- изменять системные файлы Siemens;
- писать во все обнаруженные профили;
- добавлять пути в глобальный `PATH`;
- подменять `UGII_USER_DIR`;
- редактировать бинарный `.mtx`;
- копировать Bridge DLL в `custom/startup`;
- удалять файл, которого нет в предыдущем package manifest;
- выполнять commit до проверки staging SHA-256;
- оставлять частично установленный пакет без rollback.

## Конфигурационные инварианты

- `schema_version = 3`;
- ровно 12 базовых сочетаний;
- каждый базовый `BUTTON ID` фиксирован policy;
- 14 активных модулей канонического профиля;
- восемь уникальных слотов каждого модуля;
- уникальные внутренние префиксы;
- уникальные клавиши сетки;
- точный `BUTTON ID` каждой модульной команды;
- производные последовательности не сериализуются.

## Ограничения

CI не воспроизводит фактическую чувствительность UI-команд в лицензированной среде NX. Разрушительные действия необходимо проверить на копии данных и целевой рабочей станции.
